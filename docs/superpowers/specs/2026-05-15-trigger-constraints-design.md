# 触发器级约束条件设计

日期：2026-05-15

## 背景

vFlow 当前支持触发器模块和工作流内的条件判断，但缺少 MacroDroid 类似的“触发前约束条件”。工作流里的 `If` 只能在触发器命中并启动工作流之后判断，无法阻止高耗电触发器在后台采样、扫描或监听。

典型场景是位置触发器：用户只希望在“模式不是工作”且“时间还在上午”时才允许位置触发器获取位置并判断是否进入围栏。若约束在触发前生效，位置、蓝牙、无障碍等后台触发器可以在不满足低成本条件时暂停或跳过昂贵检查，从而减少耗电和后台噪音。

## 目标

- 为每个触发器单独配置约束条件。
- 约束本身采用模块化设计，沿用触发器模块和动作模块的扩展思路。
- 第一版支持多个约束全部满足后才允许触发。
- UI 上在触发器编辑页内管理约束，避免把约束误认为工作流动作步骤。
- 对已有工作流保持兼容；没有约束的触发器行为不变。

## 非目标

- 第一版不做工作流级全局约束。
- 第一版不做嵌套条件组、OR 组合或复杂表达式编辑器。
- 第一版不允许高成本约束参与位置触发器的采样前判断，例如“先获取当前位置再判断位置约束”。
- 第一版不把约束显示为可执行工作流步骤，也不改变工作流主步骤执行顺序。

## 推荐方案

采用“触发器级约束模块”方案：每个触发器 `ActionStep` 可挂载一组约束 `ActionStep`。约束模块进入新的模块分类，例如 `constraint`，参数编辑、摘要、校验、导入导出和自定义模块扩展尽量复用现有模块基础设施。

多个约束的组合规则固定为 AND：所有约束都返回允许，触发器才继续执行或注册昂贵监听。后续可以在数据结构中加入组合模式字段，但第一版先不暴露。

## 数据模型

在 `ActionStep` 增加字段：

```kotlin
val constraints: List<ActionStep> = emptyList()
```

含义：

- 普通动作步骤可以保留空约束列表。
- 触发器步骤使用该字段存储约束模块实例。
- 约束模块仍使用 `moduleId`、`parameters`、`isDisabled` 和 `id`。
- 旧工作流反序列化时默认 `constraints = emptyList()`，无需迁移。

若后续需要 OR 或条件组，可扩展为：

```kotlin
val constraintMode: String = "all"
```

第一版不加入该字段，避免提前承诺未实现语义。

## 模块体系

新增约束模块概念：

- 约束模块属于 `ModuleCategories.CONSTRAINT`。
- 约束模块不是执行型动作步骤，不进入主工作流步骤列表。
- 约束模块需要提供可编辑参数、摘要、校验和评估逻辑。
- 自定义约束模块可以复用现有模块注册、参数和 UI 基础设施，但执行入口是约束评估器，而不是 `WorkflowExecutor`。

建议抽象：

```kotlin
interface ConstraintModule : ActionModule {
    suspend fun evaluateConstraint(context: ConstraintEvaluationContext, step: ActionStep): ConstraintResult
}
```

`ConstraintResult` 包含：

- `allowed: Boolean`
- `reason: String?`
- `cost: ConstraintCost`

`ConstraintCost` 用于区分约束是否适合触发前调度：

- `LOW`：读取时间、星期、全局变量、屏幕状态、充电状态等。
- `MEDIUM`：读取系统状态但通常不主动采样，例如当前网络状态。
- `HIGH`：需要主动传感器、定位、扫描、网络请求或无障碍查询。

第一版触发器采样前只使用 `LOW` 和明确标记可安全的 `MEDIUM` 约束。`HIGH` 约束只能作为触发命中后的最终拦截，不能用于省电暂停。

## 第一批约束模块

第一版建议实现这些低成本约束：

- 时间段约束：当前时间在/不在指定时间段。
- 星期约束：当前星期属于/不属于选中日期。
- 全局变量约束：指定全局变量与值比较，复用现有条件操作符。
- 充电状态约束：正在充电/未充电。
- 屏幕状态约束：亮屏/息屏。
- 网络状态约束：Wi-Fi、移动网络、无网络等当前连接状态。

用户示例可以表达为：

```text
位置触发器：进入公司
约束 1：全局变量 mode 不等于 工作
约束 2：当前时间在 00:00-12:00
```

## 运行时流程

触发器处理分两层：

1. 注册/采样前约束：只评估低成本约束。若不满足，则允许 handler 暂停或降级高耗电监听。
2. 命中前最终约束：触发器已经收到系统事件或位置更新后，再评估完整约束。若不满足，则记录跳过原因，不启动工作流。

位置触发器示例：

- 约束不满足时，位置 handler 不启动主动网络/GPS 监听。
- 可以保留被动监听，因为被动监听复用其他应用的位置更新，耗电很低。
- 约束重新满足时，handler 可恢复网络/GPS 策略。
- 时间段、星期等约束需要安排下一次重评估，避免只能等到下一次位置事件才恢复。

通用触发器执行入口需要在调用 `TriggerExecutionCoordinator.executeTrigger()` 前评估最终约束。这样所有触发器都获得一致的跳过语义。

## UI 设计

在触发器编辑页内新增“约束条件”折叠区，位置在触发器参数和异常处理策略之间。

内容：

- 标题：`约束条件`
- 副标题：无约束时显示 `始终允许触发`
- 有约束时显示 `需满足 N 个约束`
- 主按钮：`添加约束`
- 约束列表：每条显示模块名、摘要、启用状态和删除入口。

交互：

- 点击 `添加约束` 打开模块选择器，但只显示 `constraint` 分类。
- 点击已有约束卡片复用 `ActionEditorSheet` 编辑该约束模块参数。
- 删除约束需要确认，避免误删。
- 约束默认全部启用；禁用约束可保留配置但不参与评估。
- 多个约束第一版固定为“全部满足”。

工作流主画布的触发器摘要可以显示轻量标记：

```text
位置触发器：进入公司 · 2 个约束
```

这样用户在列表中能发现触发器带了限制，但详细管理仍留在触发器编辑页。

## 导入导出与兼容

- `ActionStep.constraints` 使用默认空列表，旧 JSON 缺省字段时行为不变。
- 导出时保留约束模块及参数。
- 导入时若遇到未知约束模块，保留原始 step 数据并在 UI 中显示“未知约束”，但评估时按不允许处理，避免误触发。
- 如果未知约束被用户删除，保存后从数据中移除。

## 错误处理与日志

约束评估失败不应启动工作流。建议规则：

- 配置无效：跳过触发并记录“约束配置无效”。
- 权限缺失：跳过触发并记录缺失权限。
- 运行异常：跳过触发并记录异常摘要。
- 约束不满足：记录调试日志，可选写入轻量执行日志，避免日志被高频触发刷屏。

触发器诊断应扩展为同时诊断约束：

- 显示每个约束当前是否满足。
- 若位置触发器因约束不满足而不会启动主动定位，诊断里明确显示。

## 测试策略

需要覆盖：

- `ActionStep` JSON 兼容：缺少 `constraints` 字段时默认空列表。
- 约束模块评估：时间段、星期、全局变量等边界。
- 触发器最终拦截：约束不满足时不调用 `WorkflowExecutor`。
- 位置触发器采样策略：低成本约束不满足时不启动主动网络/GPS 监听。
- 导入未知约束模块：保留数据但阻止触发。
- UI 状态读取保存：新增、编辑、删除、禁用约束后保存到触发器 step。

## 实施顺序建议

1. 扩展 `ActionStep` 数据模型和序列化兼容测试。
2. 增加约束模块分类、约束接口、评估上下文和评估器。
3. 实现第一批低成本约束模块。
4. 在 `TriggerExecutionCoordinator` 或 handler 调用前加入最终约束拦截。
5. 为位置触发器加入低成本约束驱动的主动监听暂停/恢复。
6. 在 `ActionEditorSheet` 增加约束折叠区和约束编辑流程。
7. 扩展触发器诊断结果，显示约束状态。

## 风险与取舍

- 将约束挂在触发器 step 上，比工作流级约束更能省电，但需要触发器编辑 UI 和 handler 层同时理解约束。
- 复用 `ActionStep` 能降低编辑器和导入导出成本，但需要防止约束模块被误插入主步骤列表或被 `WorkflowExecutor` 当作普通动作执行。
- 第一版只支持 AND 会限制表达能力，但可以覆盖大多数省电场景，并保持评估语义清晰。
- 对位置触发器来说，约束恢复需要定时重评估，否则跨过时间段后可能不能及时恢复主动定位。

## 验收标准

- 用户可以在任意自动触发器上添加、编辑、删除约束模块。
- 没有约束的触发器行为完全不变。
- 多个约束全部满足时才允许触发工作流。
- 约束不满足时，触发器不会启动工作流，并能在诊断中看到原因。
- 位置触发器在低成本约束不满足时不会继续维持主动网络/GPS 定位。
- 旧工作流可以正常打开、保存和执行。
